# Password reset and invite tokens

This folder plus a few helpers in `lib/` implements two flows that share the same machinery:

- **Reset password** — user forgot their password; clicks a link, sets a new one. Token lives 10 minutes.
- **Invite a new user** — admin creates a user; user clicks a link, picks their first password. Token lives 7 days.

Same DB table, same URL, same cookie, same form, same Server Action. Only the **creation site** and **TTL** differ. Everything below applies to both unless noted.

## What a token actually is

A token is a 64-character hex string generated by `crypto.randomBytes(32).toString('hex')`. It is **not** a password hash, not derived from any user data, not signed — it's pure random. Its only job is to be unguessable.

For every token, **two** things are stored at creation time:

1. **The raw token** — sent only in the email. Lives in: the email body, the user's cookie briefly, server memory during the request. Never persisted as plaintext.
2. **A row in `PasswordResetToken`**:

   ```
   tokenHash   = SHA256(rawToken)     ← stored; can't be reversed to the raw value
   userId      = the user this token authorizes a password change for
   expiresAt   = when the token stops working
   usedAt      = when it was consumed (null until then)
   ```

**The user-binding lives in the `userId` column, not in the token.** The token is just an unforgeable lookup key. The DB row says "whoever shows up with a token that hashes to `def456...` is authorized to change Bob's password."

> Coat-check metaphor: the DB row is the ledger entry ("ticket #def456 → Bob's coat"). The raw token is the physical ticket in the user's email. The user shows up with the ticket; the server looks up the ledger to figure out whose coat to release.

## The full flow

```
1. Token created (admin creates user, OR user requests reset)
   - rawToken  = randomBytes(32).toString('hex')         → "abc123..."
   - tokenHash = SHA256(rawToken)                         → "def456..."
   - DB row: { tokenHash="def456...", userId="bob", expiresAt=... }
   - Email sent: "Click https://.../reset-password/abc123..."

2. User clicks the link
   - GET /reset-password/abc123...
   - app/reset-password/[token]/route.ts runs
   - Format check: must be 64 hex chars (cheap regex, no DB)
   - Sets HttpOnly cookie: reset_password_token = "abc123..."
   - Redirects to /reset-password/update

3. User sees password form
   - URL bar shows /reset-password/update (no token in URL)
   - Form is generic — it does not know which user it's for

4. User submits new password
   - Server Action: resetPasswordFromCookie
   - Read cookie → "abc123..."
   - Recompute SHA256 → "def456..."
   - DB lookup: row where tokenHash="def456..." AND usedAt IS NULL
   - Row contains userId="bob" → THAT is the user to update
   - UPDATE User SET passwordHash=<new bcrypt>, sessionVersion+=1 WHERE id="bob"
   - UPDATE PasswordResetToken SET usedAt=NOW() WHERE tokenHash="def456..."
   - Clear cookie
```

## Why the cookie pattern exists

The two-step "GET sets cookie → POST consumes cookie" is not laziness, it's deliberate:

1. **Token leaves the URL fast.** After the GET redirects, the address bar shows `/reset-password/update` — the token isn't sitting in the user's URL bar to be screenshotted or accidentally shared.
2. **Email scanners can't burn the token.** Corporate mail systems (Microsoft SafeLinks, Mimecast, Proofpoint) auto-follow every link in incoming email to scan for malware. The GET only sets a cookie — actual token validation happens on POST. A scanner visit is a harmless no-op.
3. **HttpOnly cookie is XSS-resistant.** If a script ever gets injected on the password-set page, it can't read the cookie. A hidden `<input>` field would be readable by any script.
4. **The form page is static and stupid.** `/reset-password/update` doesn't need to know the token. It's a generic password-entry form with no token state in the rendered HTML.

The cookie is path-scoped to `/reset-password`, `httpOnly`, `secure` (in production), `sameSite: lax`. See `lib/reset-password.ts`.

## The two creation sites

|  | Reset flow | Invite flow |
|---|---|---|
| Trigger | User submits email on `/reset-password` | Admin submits form on `/admin/users/new` |
| Token created in | `requestPasswordReset` action | `createUser` action |
| TTL | 10 minutes | 7 days |
| User's password before | Their existing real password (still valid until reset) | A random hash nobody knows (account unreachable until activation) |
| `mustChangePassword` before | Whatever it was | `false` (no temp password to swap out) |

After consumption, both end in exactly the same DB state: a real user-chosen `passwordHash`, `mustChangePassword=false`, `sessionVersion` incremented.

## Security properties this design has

- **DB leak** → only hashes leak. An attacker would need to find a 64-byte input that SHA-256s to a stored hash. Computationally infeasible.
- **Email archive leak (later)** → tokens are expired, the URLs are dead. Useless to the attacker.
- **Email in-flight access** → if an attacker controls the inbox at the moment of delivery, they take over the account. This is fundamental to email-based reset and is not solvable here.
- **Single use** → `usedAt` is set on success. The same token cannot be reused.
- **Session invalidation** → `sessionVersion` increments on every password change; existing JWTs become invalid on the next request (see `lib/auth.ts`).
- **Format check is cheap** → the GET on the link doesn't hit the DB. Only the POST that consumes the token does. Keeps scanner traffic from causing DB load.

## Gotchas / don't-do

- **`[token]/route.ts` is a Route Handler, not a Server Component.** Next.js 16 forbids setting cookies from Server Components. Don't convert it back to a `page.tsx`.
- **The cookie path is `/reset-password`.** If you move the password-update form to a different path, update the cookie path or the cookie won't be sent.
- **Don't validate the token against the DB inside the route handler.** That would consume tokens when email scanners auto-follow the link.
- **Don't put the token in a query string** (`?token=...`). Query strings get into server access logs and Referer headers. The current design only has the token in the URL path for ~50ms, and `Referrer-Policy: no-referrer` is set on `/reset-password/*` in `next.config.ts`.
- **`PasswordResetToken.userId` is the only source of truth for who the token authorizes.** Don't add a `userId` form field that the client could control — that would let anyone reset any user's password if they knew a valid token AND a user ID.
- **The placeholder `passwordHash` on freshly-invited users is intentional, not a bug.** It's the bcrypt of random bytes the server immediately discarded. Nobody knows the plaintext. The account is unreachable by login until activation overwrites the hash. See [ADR 0001](../../docs/adr/0001-invite-link-flow.md) for the reasoning.
